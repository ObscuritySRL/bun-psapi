# PROMPT.md — Generating a Bun FFI Win32 DLL Wrapper

You are tasked with creating a complete, zero-dependency Bun FFI wrapper for a Windows DLL (e.g., `shlwapi.dll`, `advapi32.dll`, `ntdll.dll`, etc.). The wrapper must achieve **full API coverage** of every exported function in the target DLL.

## Reference Projects

Before writing any code, **read and analyze** both of these existing projects in their entirety. They are the canonical source of truth for patterns, conventions, and structure:

- `D:\Projects\bun-kernel32` — Kernel32 bindings
- `D:\Projects\bun-user32` — User32 bindings

Read every `.ts` file in each project. Pay close attention to:

- `index.ts` — entry point pattern
- `runtime/extensions.ts` — `.ptr` property installation on TypedArrays
- `types/<Name>.ts` — type aliases, enums, and constants
- `structs/<Name>.ts` — FFI symbol map, `Load`/`Preload` mechanism, public static methods
- `package.json` — metadata, exports, engines, files

Your output must be **structurally identical** to these projects, as if all three were generated by the same tool.

## Step-by-Step Process

### 1. Discover All Exports

Use `dumpbin` to enumerate every exported function from the target DLL:

```bash
cmd.exe /c "D:\Projects\bun-kernel32\bin\dumpbin.exe /exports C:\Windows\System32\<target>.dll"
```

This gives you the **authoritative, complete** list of functions to bind. Every single export must appear in the final `Symbols` map and have a corresponding public static method. Do not omit any.

### 2. Look Up Every Function Signature

For each exported function, look up the **exact** signature from the [Microsoft Win32 API documentation](https://learn.microsoft.com/en-us/windows/win32/api/). You need:

- The exact parameter names (use them verbatim)
- The exact parameter types
- The return type
- The documentation URL (for the comment above each method)

**Parameter names must match the MS documentation exactly.** If Microsoft says `hProcess`, you write `hProcess`. If they say `lpBaseName`, you write `lpBaseName`. No renaming, no camelCase conversion, no abbreviation changes.

### 3. Map Win32 Types to FFI Types

This is the most critical step. The mapping rules are:

#### Handle types → `FFIType.u64` (TypeScript: `bigint`)

Any type that represents a handle or opaque system resource:

```
HANDLE, HMODULE, HINSTANCE, HWND, HDC, HICON, HBRUSH, HBITMAP,
HKEY, HCURSOR, HMENU, HHOOK, HDESK, HMONITOR, HRGN, HACCEL,
SC_HANDLE, SERVICE_STATUS_HANDLE, etc.
```

All of these are `FFIType.u64` and typed as `bigint` in TypeScript.

#### Pointer-as-address (by value) → `FFIType.u64` (TypeScript: `bigint`)

When an `LPVOID`, `LPCVOID`, or similar pointer type is used as a **raw address value** (not a buffer to fill), it must be `FFIType.u64` and typed as plain `bigint`. This applies to:

- Memory addresses (e.g., `lpBaseAddress` in `ReadProcessMemory`)
- Allocation addresses (e.g., `lpAddress` in `VirtualAllocEx`)
- Device driver image bases (e.g., `ImageBase` in `GetDeviceDriverBaseName`)
- Mapped view addresses (e.g., `lpv` in `GetMappedFileName`)

**How to tell the difference:** If the parameter is an address the caller already has (from a previous API call, from reading a buffer of pointers, etc.) and is not a buffer the callee will read from or write into, it's an address-by-value → `u64`.

#### Pointer-as-buffer (in/out buffers) → `FFIType.ptr` (TypeScript: `Pointer`)

When a pointer parameter represents a buffer the caller allocates and the callee reads from or writes into:

```
LPVOID, LPCVOID, LPSTR, LPWSTR, LPCSTR, LPCWSTR, LPDWORD, LPBYTE,
LPOVERLAPPED, LPSECURITY_ATTRIBUTES, LP<anything>, P<anything>
```

These are `FFIType.ptr` and typed as `Pointer` (via a type alias).

#### Integer types → appropriate FFI integer type

| Win32 Type | FFI Type | TypeScript |
|---|---|---|
| `BOOL` | `FFIType.i32` | `number` |
| `DWORD`, `UINT`, `ULONG` | `FFIType.u32` | `number` |
| `INT`, `LONG` | `FFIType.i32` | `number` |
| `WORD`, `USHORT` | `FFIType.u16` | `number` |
| `SHORT` | `FFIType.i16` | `number` |
| `BYTE` | `FFIType.u8` | `number` |
| `SIZE_T`, `ULONG_PTR`, `DWORD_PTR`, `UINT_PTR` | `FFIType.u64` | `bigint` |
| `LONG_PTR`, `INT_PTR`, `SSIZE_T` | `FFIType.u64` | `bigint` |
| `LARGE_INTEGER`, `ULARGE_INTEGER`, `LONGLONG`, `ULONGLONG` | `FFIType.u64` | `bigint` |
| `VOID` (return) | `FFIType.void` | `void` |

#### Callback function pointers → `FFIType.ptr` (TypeScript: `Pointer`)

Callback typedefs like `WNDENUMPROC`, `HOOKPROC`, `LPTHREAD_START_ROUTINE`, etc. are all `FFIType.ptr`.

#### Return type conventions

- Functions returning `BOOL` → `FFIType.i32`
- Functions returning a handle → `FFIType.u64`
- Functions returning a count or length as `DWORD` → `FFIType.u32`
- Functions returning `VOID` → `FFIType.void`
- Functions returning a pointer (e.g., `LPVOID` from `VirtualAlloc`) → `FFIType.u64` (since the caller receives it as a bigint address)

### 4. Create the Project Structure

```
bun-<apiname>/
├── index.ts
├── package.json
├── tsconfig.json
├── runtime/
│   └── extensions.ts
├── types/
│   └── <ApiName>.ts
├── structs/
│   └── <ApiName>.ts
└── example/
    └── <apiname>.ts
```

Copy the exact structure from the reference projects. The naming convention:
- Directory/package: `bun-<apiname>` (lowercase)
- Class name: `<ApiName>` (PascalCase, e.g., `Shlwapi`, `Advapi32`, `Ntdll`)
- DLL name in `dlopen`: `'<apiname>.dll'`

### 5. Write Each File

#### `index.ts`
Three lines. Import extensions, import the class, re-export types. Identical pattern to kernel32/user32.

#### `runtime/extensions.ts`
**Copy verbatim** from the reference projects. This file is identical across all wrappers — it installs `.ptr` on all TypedArray prototypes.

#### `types/<ApiName>.ts`
- Import `Pointer` from `bun:ffi`
- Define **all** relevant enums with their hex values (use `0x0000_0000` underscore-separated formatting for readability, matching the reference projects)
- Define **all** type aliases needed by the API methods
- Include any constants (as `export const` objects with `as const`)
- Alphabetize within each category (enums, then types)

**Be thorough with enums.** If the API uses flag values, enumeration constants, or option sets, include them. Look at how kernel32 defines `ProcessAccessRights`, `MemoryProtection`, `FileAccess`, etc. — every relevant constant set should have an enum.

#### `structs/<ApiName>.ts`
- Import `FFIFunction`, `FFIType`, `dlopen` from `bun:ffi`
- Import all needed types from `../types/<ApiName>`
- Class with JSDoc matching the reference projects
- `private static Load<T>()` method — identical pattern
- `public static Preload()` method — identical pattern
- `private static readonly Symbols` — **alphabetized**, with `as const satisfies Record<string, FFIFunction>`
- One `public static` method per export, each with:
  - A comment containing the MS docs URL
  - Parameter names matching MS docs exactly
  - Parameter types from the type aliases
  - A single-line body: `return <Class>.Load('<FunctionName>')(params...);`

#### `package.json`
Follow the kernel32/user32 pattern. Update:
- `name`: `bun-<apiname>`
- `description`: `Zero-dependency, zero-overhead Win32 <APINAME> bindings for Bun (FFI) on Windows.`
- `keywords`: include the API name
- `homepage`/`repository`/`bugs`: update URLs

#### `tsconfig.json`
Copy from the reference projects. Use the kernel32 version (it has `@types/*` path mapping).

### 6. Verify

1. **Type-check:** `bunx tsc --noEmit` must pass with zero errors.
2. **Runtime test:** Write an example script that exercises at least 2-3 core functions and confirm they return sensible values.
3. **Export count:** Verify the number of public static methods matches the number of exports from `dumpbin`.

## Common Pitfalls

- **Don't confuse pointer-as-buffer with pointer-as-address.** A `LPVOID lpAddress` that represents "where to allocate memory" is `u64` (bigint). A `LPVOID lpBuffer` that represents "write output here" is `ptr` (Pointer). When in doubt, look at kernel32's treatment of `ReadProcessMemory`, `WriteProcessMemory`, `VirtualAllocEx`.
- **BOOL returns are `i32`, not `u32`.** Win32 BOOL is a signed 32-bit int.
- **DWORD returns are `u32`.** Functions that return a character count, byte count, or error code as DWORD use `u32`.
- **All handle types are `u64`, never `ptr`.** Even though handles are technically pointers, Bun FFI requires them as bigint.
- **Symbols must be alphabetized.** This is a hard rule across all three reference projects.
- **Wide string variants (W suffix) take `LPWSTR`/`LPCWSTR` which are `FFIType.ptr`.** The caller is responsible for encoding to UTF-16LE with null terminator.
- **Include both A and W variants** of string functions if the DLL exports them. Don't skip the ANSI versions.
- **The `as const satisfies Record<string, FFIFunction>` on the Symbols object is required.** It enables type inference for the Load method.

## Checklist

Before declaring the wrapper complete:

- [ ] Every export from `dumpbin` has a Symbol entry and a public static method
- [ ] All parameter names match MS documentation verbatim
- [ ] All FFI types are correct (`u64` for handles/addresses, `ptr` for buffers, correct integer widths)
- [ ] Symbols are alphabetized
- [ ] All relevant enums and constants are defined in the types file
- [ ] `bunx tsc --noEmit` passes cleanly
- [ ] At least one runtime smoke test confirms bindings load and return correct data
- [ ] File structure is identical to bun-kernel32 and bun-user32
